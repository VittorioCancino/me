From Model To Language

Why Do We Need SQL?

Last time we studied how queries work conceptually. Now we need the language that real database systems understand.

A familiar situation Imagine a coordinator asks: "Which first-year students are taking Databases?" We already understand the logic behind that question. We may need to pick columns, filter rows, and combine tables. What we still need is the language that turns that idea into something a real database can execute.
The Path

From Question To Answer

SQL is the bridge between a real question and the result we want to read.

→
Language Write it in SQL

→

Let the system compute the rows we want.

Core Idea

SQL In One Sentence

SQL is a declarative language for working with relational databases.

SQL is a declarative language for working with relational databases.

Declarative means

We describe what we want from the data instead of every low-level step the computer should follow.

Relational means

Data is organized into structured tables. Those tables are connected through relationships.

For today: SQL can also create, insert, update, and delete data, but in this practicum we focus on querying with SELECT.

The Running Example Schema

`students` stores who the students are. `courses` stores the courses that exist. `enrollments` connects both relations and tells us which student takes which course. In the next slides, we will populate these relations with example rows and start querying them.

Start With Questions

What Kind of Questions Do We Want To Ask?

Before learning syntax, think about the questions we can already ask with the sample rows on the previous slide.

Questions from one relation

  • What are the names of all students in `students`?
  • Which students are in year 1?
  • Which courses have 4 credits?
  • What grade does student 101 have?

Questions across relations

  • Which students are enrolled in Databases?
  • Which course is Ana taking?
  • Which grades belong to first-year CS students?

Once a question needs names from `students` and course titles from `courses`, one table is no longer enough.

First Syntax

The Shape of a First Query

A basic query starts by saying what we want to see and where the data comes from.

Suppose we want to answer the question: "What are the names and majors of all students?"

Example query Read-only
SELECT name, major
FROM students;

This is the first shape to remember: choose the columns you want, then say which relation they come from.

How To Read It

Reading the Query Step by Step

Once the query is written, we can explain what each part is doing.

SELECT

Tells us which columns we want to see in the result.

FROM

Tells us which relation we are reading from.

Result

The output is still a table-shaped result, even if it is not stored permanently.

First Result

What Does This Query Return?

The query reads `students` and returns only the columns we asked for.

result
name major
Ana CS
Lucas Business
Sofia CS

Notice that `student_id` and `year` do not appear here. They exist in the original relation, but we did not ask for them in `SELECT`.

Filtering Rows

When We Want Only Some Rows

`SELECT` chooses columns. `WHERE` lets us filter which rows stay in the result.

Suppose we want to answer: "Which students are in year 1?"

Example query Read-only
SELECT name, year
FROM students
WHERE year = 1;

This adds one new idea: after choosing the relation, we can keep only the rows that satisfy a condition.

How To Read WHERE

Reading a Query With WHERE

We can now read the query as a short sequence of decisions.

FROM

Start from the `students` relation.

WHERE

Keep only the rows where `year = 1`.

SELECT

Show only the `name` and `year` columns from those remaining rows.

RESULT

The output now includes Ana and Sofia, because they are the students in first year.

More Precise Filters

Using AND

`AND` helps when all conditions must be true at the same time.

AND

Use `AND` when both conditions must be true.

SELECT name
FROM students
WHERE year = 1 AND major = 'CS';
More Precise Filters

Using OR

`OR` helps when more than one possible condition can match the rows we want.

OR

Use `OR` when at least one condition can be true.

SELECT title
FROM courses
WHERE credits = 4 OR credits = 5;
Presentation Matters

Ordering Results with ORDER BY

Even if the right rows are returned, we may want them in a meaningful order.

If we list students by name, the result is easier to read than if rows appear in an arbitrary order.
Example query Read-only
SELECT name, major
FROM students
ORDER BY name ASC;
ASC

Ascending order: A to Z, or smaller values to larger values.

DESC

Descending order: Z to A, or larger values to smaller values.

A Difference From Relational Algebra

SQL Does Not Remove Duplicates By Default

This is one important difference between relational algebra and SQL.

In relational algebra, we usually think in terms of sets, so duplicate tuples disappear automatically. In SQL, query results can contain repeated rows unless we explicitly remove them.
R.A.

Results behave like sets, so duplicates are not kept.

SQL

Results can keep repeated values unless we ask for uniqueness.

A Concrete Example

A Query That Produces Duplicates

If we project only one column, repeated values may appear in the result.

Suppose we ask: "What majors exist among our students?"

Example query Read-only
SELECT major
FROM students;

This query is valid, but because two students share the same major, the result can contain repeated values.

The Result

Why Do Duplicates Appear?

Once we remove identifying columns, repeated values can become visible.

result
major
CS
Business
CS

`CS` appears twice because two different students share that same value. We removed `student_id` and `name`, so the repeated major is now visible.

Unique Values

Using DISTINCT

When we want unique values only, SQL gives us `DISTINCT`.

Example query Read-only
SELECT DISTINCT major
FROM students;

`DISTINCT` removes repeated rows from the final result. In this case, each major appears only once.

The Result

What Does DISTINCT Return?

The repeated `CS` value disappears, so each major appears only once.

result
major
CS
Business

This is the same query result after SQL removes duplicate rows from the final output.

A New Kind Of Question

A Question One Relation Cannot Answer

Up to now, we have worked with one relation at a time. Now we want a richer answer.

Suppose we ask: "Which students are enrolled in Databases?" The answer clearly exists in the database, but it does not live inside just one relation.

This is the moment where SQL has to combine information from multiple relations.

Locate The Information

Why One Relation Is Not Enough

Each relation gives us part of the answer, but not the whole answer.

students

Gives us the student names, but not the course titles they take.

courses

Gives us course titles like `Databases`, but not the names of enrolled students.

enrollments

Connects `student_id` and `course_id`, but stores ids instead of the descriptive names we want to show.

The Core Idea

We Need To Combine Relations

A join matches rows that belong together through shared attributes.

Step 1

Match `students.student_id` with `enrollments.student_id`.

Step 2

Match `enrollments.course_id` with `courses.course_id`.

Result

Now student names and course titles can appear together in the same query result.

First JOIN

A First JOIN Query

Here is the basic shape of a query that combines multiple relations.

Example query Read-only
SELECT students.name, courses.title
FROM students
JOIN enrollments ON students.student_id = enrollments.student_id
JOIN courses ON enrollments.course_id = courses.course_id;

`JOIN ... ON ...` tells SQL exactly which rows should be matched together.

How To Read It

Reading the JOIN Step by Step

The query becomes easier once we explain one clause at a time.

FROM

Start from the `students` relation.

JOIN

Add `enrollments` by matching the same `student_id` values.

JOIN

Add `courses` by matching the same `course_id` values.

SELECT

Show the student name together with the course title.

The Payoff

Now We Can Answer the Original Question

Once the relations are joined, `WHERE` can focus on the exact course we care about.

Example query Read-only
SELECT students.name
FROM students
JOIN enrollments ON students.student_id = enrollments.student_id
JOIN courses ON enrollments.course_id = courses.course_id
WHERE courses.title = 'Databases';
The Result

Which Students Are Enrolled in Databases?

The final result contains the names of the students enrolled in that course.

result
name
Ana
Sofia

A New Kind Of Question

Sometimes We Need Summaries, Not Rows

So far, our queries returned individual rows. Now we want answers that summarize the data.

Questions like "How many students are there?" or "What is the average grade?" do not ask for tuples one by one. They ask SQL to compute a single summary value from many rows.
Row questions

Return tuples such as names, titles, or grades.

Summary questions

Return values such as counts, averages, totals, minima, or maxima.

How Aggregates Read Data

First SQL Determines Which Rows Are In Scope

Before an aggregate can count or average anything, SQL must first determine which rows the query is talking about.

`FROM` chooses the relation we start from
`JOIN` adds matching rows from other relations
`WHERE` removes rows that do not satisfy the condition
The result is a temporary set of rows produced by the query so far. This is the set that aggregate functions summarize.
Intermediate Result

Think Of It As A Working Table

After `FROM`, `JOIN`, and `WHERE`, SQL has rows available to work with, even if that table is not shown directly to us.

Example query Read-only
SELECT AVG(grade)
FROM students
WHERE grade >= 10;

We do not average the whole database. We average the rows that remain after the condition is applied.

working rows
student grade
Ana 16
Sofia 14
Leo 18
Summary Step

Then The Aggregate Reads That Working Table

An aggregate function looks at the rows in the working table and produces one summary value.

`COUNT(*)`

Counts how many rows are in the working table.

3 rows -> 3
`AVG(grade)`

Uses the values from one column in those rows.

(16 + 14 + 18) / 3 -> 16
Aggregate functions do not work on the raw database directly. They work on the rows produced by the query so far.
Aggregate Functions

The Basic Aggregate Functions

Aggregate functions combine many rows into one result value.

COUNT(*)

Counts how many rows are in the result.

AVG(column)

Computes the average of a numeric column.

SUM(column)

Adds together the numeric values of a column.

MIN / MAX

Return the smallest or largest value in a column.

First Summary Query

A First Aggregate Query

A simple aggregate query asks for one summary value over a relation.

Suppose we ask: "How many students are in the relation?"

Example query Read-only
SELECT COUNT(*)
FROM students;
result
count
3
Filtering Before Aggregating

Aggregation After Filtering

`WHERE` still works before the aggregate is computed.

Suppose we ask: "How many first-year students are there?"

Example query Read-only
SELECT COUNT(*)
FROM students
WHERE year = 1;
result
count
2
Grouping Rows

What If We Want One Summary Per Group?

Sometimes one total is not enough. We may want one summary for each category of rows.

For example, instead of asking "How many students are there?", we may ask "How many students are there in each year?" That means SQL must first divide the rows into groups and then summarize each group.
One total

`COUNT(*)` over the whole relation gives one answer.

Many totals

`GROUP BY` gives one answer for each distinct value of the grouping column.

Grouping Rows

Start With The Working Table As Usual

Before grouping, SQL still has a normal working table with one row per tuple in the current result.

working rows
year student course
1 Ana Programming
1 Leo Databases
1 Mia Math
2 Sofia Networks
2 Omar Databases

Notice that the value in `year` repeats across several rows. That repeated value is exactly what `GROUP BY year` will use to divide the table.

Grouping Rows

`GROUP BY` Divides The Working Table Into Subgroups

Rows with the same value in the grouping column belong to the same group. Then the aggregate function runs once for each group.

working rows
year student course
1 Ana Programming
Leo Databases
Mia Math
2 Sofia Networks
Omar Databases

Here we grouped by `year`. The blank cells help us visualize that consecutive rows belong to the same subgroup.

What SQL is doing
`GROUP BY year` separates the rows into:
Group 1: all rows where `year = 1`
Group 2: all rows where `year = 2`
Then an aggregate such as `COUNT(*)` or `AVG(grade)` runs over each subgroup independently.
The final result contains one row per existing group, not one row per original tuple.
First GROUP BY

A First GROUP BY Query

Using the same `year` values from the working table, `GROUP BY` creates one count for each existing group.

Example query Read-only
SELECT year, COUNT(*)
FROM students
GROUP BY year;
result
year count
1 3
2 2
How To Read It

Reading GROUP BY Step by Step

The grouped query becomes easier when we read it as a process.

From

Start from all rows in the `students` relation.

Group By

Separate the rows into groups that share the same `year` value.

Count

Count how many rows are inside each of those groups.

Result

Return one row per year, together with the number of students in that year.

Filtering Groups

Filtering Groups with HAVING

After grouping, we may want to keep only some of the groups.

Where

Filters rows before the groups are formed.

Having

Filters groups after the aggregation has been computed.

Example query Read-only
SELECT year, COUNT(*)
FROM students
GROUP BY year
HAVING COUNT(*) > 1;
Putting It Together

WHERE, GROUP BY, HAVING, and ORDER BY

A full aggregation query often combines all these ideas in one pipeline.

Example query Read-only
SELECT course_id, AVG(grade)
FROM enrollments
WHERE grade >= 6.0
GROUP BY course_id
HAVING COUNT(*) > 0
ORDER BY AVG(grade) DESC;
Putting It Together

What Each Part Of The Query Does

We can read the same query as a sequence of operations over the data.

Where

Keeps only rows with grades at least 6.0 before grouping.

Group By

Builds one group for each course.

Having

Keeps only groups that satisfy an aggregate condition.

Order By

Sorts the grouped results after the averages are computed.

Mental Model

Do Not Read SQL Only From Top To Bottom

Beginners often read the query in written order, but the easiest mental model is to follow how the DBMS builds the result step by step.

The query starts by finding rows, then possibly filters, then maybe groups, then computes summaries, and only at the end decides what to show.
Written order
`SELECT` -> `FROM` -> `WHERE` -> `GROUP BY` -> `HAVING` -> `ORDER BY`
Mental order
`FROM` -> `WHERE` -> `GROUP BY` -> `HAVING` -> `SELECT` -> `ORDER BY`
Mental Model

Build A Pipeline In Your Head

When you see a query, imagine the data moving through these stages.

1. FROM / JOIN build the working table -> 2. WHERE remove rows that do not match -> 3. GROUP BY divide the rows into groups -> 4. HAVING + aggregates compute summaries and keep the groups we want -> 5. SELECT choose what appears in the result -> 6. ORDER BY sort the final result for presentation
Mental Model

Ask These Questions In This Order

These questions help you decode almost any beginner SQL query.

What rows start the process? Look at `FROM` and `JOIN`. -> Which rows survive? Look at `WHERE`. -> Are rows grouped? Look for `GROUP BY`. -> Are groups filtered? Look for `HAVING`. -> What is finally shown? Look at `SELECT`, then `ORDER BY` if the result is sorted.
Mental Model

A Good Beginner Habit

Before solving exercises, pause and translate the query into a data transformation story.

Instead of saying "I read `SELECT` first because it is at the top", say "The DBMS starts from these rows, filters them, maybe groups them, computes summaries, and only then shows the final columns."
Query text -> working rows -> filtered rows -> groups -> summaries -> final result
