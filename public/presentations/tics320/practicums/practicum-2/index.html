A Question

How Do We Tell Rows Apart?

Once we transform concepts into tables, a new problem appears.

Imagine two rows in Product with the same name, or two rows in Production that involve the same company but different products. If we want to update, delete, or reference exactly one row, we need a way to say: this one, and no other.
That is the job of the Primary Key
⚠️ Without a reliable key, two rows can become indistinguishable.
Primary Keys
No two rows can share the same Primary Key

A primary key uniquely identifies each tuple, that is, each row in a relation.

Entity Table

The key usually comes from the entity key.

Product → id

Relationship Table

The key often comes from a combination of keys.

Production → (product_id, company_id)

→ For entity tables, the key comes from the entity. For relationship tables, it can come from the combination of participating keys.
Foreign Keys

Tables Must Still Know Each Other

A foreign key references a primary key in another table. That is how the relationship survives after the ER diagram becomes tables.

Product(id, name, category, price)
Company(id, stock_value)
Foreign Keys
Production(product_id, company_id, since)
🔗 In the generated relation, product_id points to Product and company_id points to Company.
Why They Matter
Foreign Keys Protect
The Meaning Of The Data
1 They maintain consistency between tables.
2 They enforce relationships instead of leaving them implicit.
Without foreign keys, Production could contain a product_id that does not exist in Product, or a company_id that points nowhere. The row would still be stored, but the relationship would already be broken.

Beyond Basic Transformation

Not all ER models fit the simple rules we have used so far.

Real models include:

  • Additional tables

Special Case 1
One Attribute
Many Values

This is the first case where the basic transformation rule is not enough.

Suppose a person can have several professions. That means one attribute does not store one value anymore. It stores many. If we force those values into extra columns, the schema starts guessing how many professions a person is allowed to have.
✗ Incorrect: create multiple columns for the same idea.
✓ Correct: create a new table.
Professions(person_id, profession)

Each profession becomes a separate row.

Why The Shortcut Fails
Why Not
Multiple Columns?
1 It produces null values.
2 It limits the number of values in advance.
3 It breaks relational design principles.
If a person can have 1 profession, 2 professions, or 10 professions, the schema should not guess that number in advance. The relational model handles repeating values by creating more rows, not more columns.
Special Case 2
A Weak Entity
Cannot Stand Alone

It depends on another entity to become identifiable.

Think about an Evaluation inside a Course.
Midterm is not enough by itself.
It only becomes meaningful once we know which course it belongs to.
Evaluation Depends On Course

A weak entity cannot be uniquely identified alone.

Transformation Rule
To Store A Weak Entity,
We Carry Two Keys

The relation must include the weak entity's partial key and the key of its parent.

  • 1 Create a table for the weak entity.
  • 2 Include its partial key and the parent key.
Evaluation(name, course_id, date)

Primary key: (name, course_id)

The row becomes identifiable only when the weak entity's own identifier is combined with the parent key.
Special Case 3
The Same Entity Can Appear
In Two Roles

Sometimes one entity participates more than once in the same relationship.

Imagine a social network where a User follows another User. It is still one entity type, but it appears twice in the relationship with different meanings.
⚠️ Problem: same entity, different roles.
Design Decision

Handling Roles

The relationship is stored once, but each participation must be named according to its role.

Follows(follower_id, followed_id, since)

Role 1

follower_id identifies the user who follows.

Role 2

followed_id identifies the user being followed.

→ Both attributes reference the same table, but they represent different roles.
Design Shortcut
Not Every Relationship
Needs Its Own Table

Sometimes the relationship can be simplified instead of being kept as a separate relation.

This happens when the multiplicity includes 1 and one entity determines the other. In that case, the relationship does not always need its own table.
  • When the multiplicity includes 1.
  • When one entity determines the other.

Example Of Simplification

Instead of keeping the relationship as a separate table, we can sometimes move the foreign key into one entity.

The foreign key moves into the entity table.

The result is a schema with fewer tables and a simpler structure.
Trade-off
Simplification Is
A Design Decision

The schema becomes easier to read, but the choice also changes what the model can express later.

+ Advantages
Simpler schema
Fewer joins
− Disadvantages
Less flexibility
May restrict future changes
Simplifying a relationship is not just a technical shortcut. It is a modeling choice, and that choice should match the meaning of the domain.
Special Case 4
One Entity Can Split
Into Subtypes

In an IsA hierarchy, a general entity can have more specific versions.

For example, an Athlete can become a Football Player or a Tennis Player. The subclasses inherit the common attributes of the parent and add their own specific properties.
Athlete → Football Player, Tennis Player

Subclasses inherit shared attributes and add their own.

Two Paths
There Is More Than
One Valid Mapping

Hierarchies are not translated with a single universal rule.

Sometimes we keep the parent table and add one table for each subclass. Other times we skip the parent table and keep only the subclass tables, each one carrying all the attributes it needs.
  • Parent table plus one table per subclass.
  • Only subclass tables, each with all attributes.
Design Choice
The Right Transformation
Depends On The Case

The hierarchy design should reflect both the semantics of the model and the way the data will be used.

  • Whether all entities belong to a subclass.
  • The amount of overlap between subclasses.
  • The type of queries the system will run.
Choose the design that feels most natural for the domain and most efficient for the queries you expect to make.

The Transformation
Is No Longer Mechanical

At this point, translating the ER model is not just applying rules.

We now need to understand the structure of the model, recognize the special cases, and make design decisions that preserve meaning without creating unnecessary complexity.

  • Design decisions during translation.
The goal is a schema that is clean, consistent, and flexible.

Relational Algebra
Relational Algebra
Begins Here

We already have structure. Now we need a formal way to query and manipulate data.

Until now, we focused on designing the relational schema. The next step is learning how to work with it: how to select, combine, and transform relations to obtain information.
Relational algebra gives us a set of operations over tables
Basic Concept
What Is
A Relation

In the relational model, a relation is the basic object we operate on.

We already met this idea when we transformed the ER model into the relational model. At that point, we were already thinking about relations as tables.
A relation is a table. It consists of tuples, which are the rows, and attributes, which are the columns.
Basic Concept
Relations Have
Set Semantics

A relation is not just a table on screen. It follows precise structural rules.

  • Relations are sets
  • There are no duplicate tuples
  • Order does not matter
These properties matter because relational algebra works with collections of facts, not with ordered lists.
Example
A Relation Can Be Written
As A Schema

We often describe a relation first by its structure, and then by the tuples that belong to it.

Student(student_id, name, major, year)
This notation gives the name of the relation and its attributes. After that, the tuples are just the specific rows that satisfy this structure.

Union Combines
Tuples

Union does not stack columns. It combines the tuples of two compatible relations into one result.

To apply union, the two relations must be union-compatible: they need the same number of attributes, and the corresponding attributes must represent compatible domains.
  • •Same number of columns
  • •Compatible attribute types
  • •Result keeps all unique tuples

Difference Keeps

Difference compares two compatible relations and keeps only the tuples that are in the first one but not in the second.

If we write R − S, the meaning is very precise: keep the tuples that belong to R and remove every tuple that also appears in S.
→ The order matters: R − S is not the same as S − R.

Composition
Operations Can Be
Composed

The output of one operation can immediately become the input of the next.

πname(σprice > 100(Product))
First we filter the products whose price > 100. Then we project only the name attribute from that result.
Meaning

Select products with price > 100, then return only their names.

Composition
A Query Can Combine
Multiple Operations

We can join first, then apply another operation over the resulting relation.

σprice > 100(Product ⨝ Factory)
We first join the two relations to connect their data. Then we filter the joined result to keep only the tuples where price > 100.
Meaning

Join the tables, then filter the resulting tuples.
