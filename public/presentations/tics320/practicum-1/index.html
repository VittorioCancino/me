A story

Meet EduTrack

An online learning startup. Ambitious team, modern stack, Y Combinator dreams.

Month 1 — Launch 200 users. Everything works perfectly. The team celebrates.
Month 4 — Growth 1,000 users. A bit slow, but manageable. They add more servers.
Month 8 — Viral moment Featured in a newsletter. 5,000 users sign up in a week.
Month 8 + 3 days — Chaos Support tickets flood in. Something is very, very wrong.
What happened

The support tickets

📋

Wrong grades

"My transcript shows a course I never took — and I'm missing one I did."

1,265 tickets in 72 hours. The CEO is on the phone. The lead dev hasn't slept.

Stop and think

The team panics. They start looking for the bug.

"It must be the payment service — let's roll back."
"No, it's the enrollment API — I see an error in the logs."
"Maybe it's the load balancer dropping requests?"
Three engineers. Three different answers. All wrong.

What actually failed?

The backend?  ·  The frontend?  ·  The infrastructure?

None of the above.

The data.
EduTrack stored users, enrollments, and payments all in one table. When two users shared an email, the system merged their records. When a course was updated, old enrollment rows silently pointed to nothing. There was no foreign key, no constraint, no single source of truth.

The code was fine. The data was broken from day one.
Key message

Data is the source of truth

  • 🔁Code can be rewritten  fixable
  • 🎨UI can be redesigned  fixable
  • ☁️Infrastructure can be replaced  fixable
  • 💾Corrupted data  stays broken

EduTrack spent 3 weeks manually reconciling 5,000 user records.
Two engineers quit. 875 customers filed chargebacks.

Your responsibility

Even if you "don't like databases".

As engineers,
you own the data.

Your users trust you with

Your users hand you their identity, their money, and their academic records. When your data model breaks, it doesn't just crash an app — it destroys trust that took years to earn.

The real cost

  • Live data migrations are slow and risky
  • Cleaning corrupted records takes weeks
  • Lost customers rarely come back

Great design is cheap. Bad design is permanent.

The verdict
Technically working.
Logically broken.
Update anomaly One change requires many updates. Miss one → corrupt data.
Insert anomaly Can't store a fact without an unrelated fact existing first.
Delete anomaly Removing one thing accidentally destroys another.
→So how do we design this so reality doesn't break our system?
One mistake
We didn't design the data —
we just stored it.

All three anomalies. Same root cause.

So instead of fixing tables…
we need to change how we think.
The shift

Think in entities, not tables

Start by asking

  • Who exists? → entities
  • What do they have? → attributes
  • How do they relate? → relationships

In EduTrack

  • User — exists independently
  • Course — exists independently
  • Instructor — exists independently
  • Enrollment — connects them
💡 Tables are an implementation detail. The real design happens before that.
The solution

This is exactly what ER design helps us do.

Bad thinking

"What columns do I need?"

Good thinking

"What objects exist in my system?"

→ ER diagrams make you think in objects and relationships — before a single table is created.
90% of a great database
is great design.

EduTrack's problem wasn't a missing index or a slow query.
It was a model that was never designed — just grown.

Today, we learn the tools to design before we build:

The tool

The Entity–Relationship Model

A way to represent the real world before turning it into tables.

🗺 A blueprint Design the structure before writing a single CREATE TABLE.
🗣 A shared language Developers, clients, and managers can all read the same diagram.
→ 3 core components: Entities, Attributes, and Relationships.

Component 2

Attributes

The fields on each paper — what describes an entity.

Types of attributes

  • Key — uniquely identifies every instance
  • Simple — atomic, indivisible (email, grade)
  • Composite — can be split (full_name → first + last)
  • Derived — computed from another (age from birthdate)
  • Multi-valued — holds several values (phone numbers)

Component 5

Participation Constraints

Can this thing exist without that relationship?

🔒 Total (mandatory)

Every instance must participate. No exceptions.

  • Every Enrollment must have a Student
  • Every Course must have an Instructor

"Can an enrollment exist without a student?" — No.

🔓 Partial (optional)

Some instances may not participate. That's fine.

  • A Student may not be enrolled right now
  • An Instructor may not be teaching this semester

"Can a student exist without enrolling?" — Yes.

→ No = Total participation. Yes = Partial participation.

Before you start
There's no single right answer.

ER modeling is a design discipline, not a formula.

Design follows functionality Every ER decision should serve a real use case — not aesthetics or habit.
Many solutions, one problem Two valid diagrams can represent the same reality differently. Both can be correct.
Practice builds judgment The only way to tell which design is better is experience, constraints, and critical thinking.
💡 The goal is not to find the perfect diagram. It's to build the right one for the right problem.
The exercise
🎬 MarBrisa Disco

You've been hired to design the database system for a disco called MarBrisa.

The situation

🌊 MarBrisa is a popular disco that hosts weekend parties with themed nights.
🎟️ They sell tickets online in advance and at the entrance on the night of each event.
🍹 Inside, there is a bar where customers can order drinks from a menu throughout the night.
→ Your job: model this system as an ER diagram. Think carefully before drawing.
What it needs to handle

Four things the system tracks

🎟️ Ticket Sales
  • Sold online (advance) or at the door
  • Each ticket has a price and belongs to an event
  • A customer can buy multiple tickets
🍹 Bar & Drinks
  • The disco has a menu of drinks
  • Each drink has a name and a price
  • Customers can order multiple drinks per night
🧾 Purchases
  • Each purchase belongs to a customer
  • Can include multiple drinks
  • Recorded with a timestamp
🎉 Events
  • MarBrisa hosts themed parties (e.g. Reggaeton Night)
  • Each event has a date
  • Ticket sales are tied to a specific event
Design challenge

Design an ER model for MarBrisa.

Guide yourself with these questions:

❓Who are the entities?
❓What attributes does each one have?
❓How do the entities relate to each other?
❓Can something exist independently, or does it always need another entity?
🔥 Optional challenge: What if a customer buys drinks for friends? Do tickets belong to a person or just to a purchase?

Don't think in tables. Think in reality.
